# Migration map — WSM (MongoDB) → PG (Postgres)

Canonical field-by-field mapping for importing legacy Workshop17 Mongo data into Proximity Green. Kept in sync with schema migrations under `packages/database/migrations/`.

Naming conventions (from `CONVENTIONS.md`):

- `snake_case`, plural table names
- `<verb>_at timestamptz` for every timestamp (no `_date` suffix ever)
- `accounting_*` prefix = definitional/identity fields that go to the accounting provider
- `commercial_*` prefix = money/terms fields
- `delivery_*` prefix = courier / physical visit address
- `external_<role>_<entity>_id` for foreign-system IDs (provider-agnostic; never `xero_id`)
- Derivable fields are dropped (licence determines membership + location)
- Mongo cruft (`__v`, `_owner_id`, `_deleted`, `createdAt`, `updatedAt`) replaced by `change_log` + `created_at` / `updated_at`

---

## Subscription model — the spine

The heart of the commercial system is this chain:

```
items (catalog)
   └─► subscription_lines (mini-contract: option → pending → signed → ended)
           ├─► subscription_line_rules (escalation / discount / surcharge / rebate)
           ├─► subscription_line_rate_history (immutable application log)
           ├─► signatures (tamper-evident capture)
           └─► invoice_lines (generated by invoice run)

licenses (access-right instance) ◀── 1:1 ── subscription_lines (when licence-backed)
contracts (umbrella legal doc)  ◀── M:N ── subscription_lines
```

**Core invariants (DB-enforced):**

1. `subscription_lines.item_id` XOR `license_id` — exactly one, never both
2. `invoice_lines.subscription_line_id` XOR `item_id` — exactly one, never both
3. `item_types.requires_license` → cannot be sold directly on an invoice or a sub by item_id (trigger)
4. `item_types.sellable_ad_hoc` / `sellable_recurring` → trigger gates which FKs are allowed
5. Sub cannot move `pending → signed` without a `signatures` row
6. Exactly one `is_primary = true` per org in `organisation_accounting_customers`

## Lineitems (WSM `lineitem`) → PG `subscription_lines` + `invoice_lines`

**Split into two tables** based on recurring vs one-off:

- **Recurring** (has `license_id` or recurring product): → `subscription_lines`

## Multi-tenant accounting: `organisation_accounting_customers`

New table — one row per (organisation, accounting tenant). Data proved 7 distinct Xero tenants across 23 locations → an org with presence in multiple jurisdictions has multiple customer records.

```
(organisation_id, provider, accounting_external_tenant_id) UNIQUE
  → accounting_external_customer_id
  → accounting_external_customer_code
  → is_primary (exactly one per org via partial unique index)
```
